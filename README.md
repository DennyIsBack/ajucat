# AJUCAT — Projeto Institucional Completo

Este repositório contém o código-fonte completo do site institucional e sistema de gerenciamento da **AJUCAT (Associação dos Juízes Católicos)**. O projeto foi desenvolvido seguindo as mais altas práticas de engenharia de software, com foco em segurança, escalabilidade e manutenibilidade.

O sistema é composto por um **frontend em Next.js 14+** e um **backend em NestJS**, conteinerizados com **Docker** e prontos para deploy em nuvem (AWS, GCP).

## Visão Geral

O projeto visa atender a dois públicos principais:

1.  **Visitantes:** Acesso a uma área pública com informações institucionais, notícias e página de contato.
2.  **Membros:** Acesso a uma área restrita (dashboard) com login, onde podem visualizar circulares, documentos internos e gerenciar conteúdo, dependendo do seu nível de permissão (Role-Based Access Control).

## Estrutura do Projeto

O projeto está organizado em uma estrutura monorepo com dois pacotes principais:

```
/ajucat
├── backend/         # Projeto NestJS (API)
├── frontend/        # Projeto Next.js (Interface)
├── nginx/           # Configuração do Nginx para produção
├── .env             # Variáveis de ambiente globais para Docker
├── .gitignore       # Arquivos ignorados pelo Git
├── docker-compose.yml # Configuração principal do Docker
├── docker-compose.dev.yml # Configuração de desenvolvimento
├── docker-compose.prod.yml # Configuração de produção com Nginx
└── README.md        # Este arquivo
```

### Backend (NestJS)

Localizado em `/backend`, segue uma arquitetura modular para separação de responsabilidades:

-   **`src/auth`**: Autenticação (login, logout, refresh token) e estratégias JWT.
-   **`src/users`**: Gerenciamento de usuários.
-   **`src/roles`**: Gerenciamento de perfis e permissões (RBAC).
-   **`src/news`**: CRUD de notícias.
-   **`src/documents`**: CRUD de documentos restritos.
-   **`src/circulars`**: CRUD de circulares internas.
-   **`src/storage`**: Abstração para upload de arquivos (Cloudinary em produção, Local para desenvolvimento).
-   **`src/audit`**: Logs de auditoria para ações críticas.
-   **`src/common`**: Módulos, guards, interceptors e filtros globais.
-   **`src/prisma`**: Schema, seed e serviço do Prisma ORM.

### Frontend (Next.js)

Localizado em `/frontend`, utiliza o App Router do Next.js 14+:

-   **`src/app/(public)`**: Rotas públicas (Home, Sobre, Notícias, Contato).
-   **`src/app/(auth)`**: Rotas de autenticação (Login).
-   **`src/app/dashboard`**: Rotas protegidas da área de membros.
-   **`src/components`**: Componentes React reutilizáveis (UI, Layout, etc.).
-   **`src/services`**: Serviços para comunicação com a API (Axios configurado).
-   **`src/context`**: Contexto de autenticação global.
-   **`src/hooks`**: Hooks customizados (ex: `useAuth`).
-   **`src/lib`**: Funções utilitárias.
-   **`src/middleware.ts`**: Middleware para proteção de rotas.

## Como Executar o Projeto

### Pré-requisitos

-   [Docker](https://www.docker.com/get-started)
-   [Docker Compose](https://docs.docker.com/compose/install/)

### 1. Configuração do Ambiente

Clone o repositório e navegue até a pasta raiz `ajucat`.

Crie uma cópia do arquivo de exemplo `.env`:

```bash
cp .env .env
```

**Importante:** Altere as variáveis de ambiente no arquivo `.env`, especialmente `JWT_SECRET` e `JWT_REFRESH_SECRET`, para valores seguros e longos em um ambiente de produção. Para upload de arquivos em produção, configure `STORAGE_PROVIDER=cloudinary` e as credenciais do Cloudinary (`CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`) — veja `backend/.env.example`.

### 2. Ambiente de Desenvolvimento

Para iniciar o ambiente de desenvolvimento com hot-reload para frontend e backend, execute:

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build
```

-   **Frontend (Next.js)** estará disponível em [http://localhost:3000](http://localhost:3000)
-   **Backend (NestJS)** estará disponível em [http://localhost:3001](http://localhost:3001)
-   **PostgreSQL** estará acessível na porta `5432`.

O banco de dados será populado automaticamente (seeding) com um usuário **ADMIN** padrão:

-   **E-mail:** `admin@ajucat.org.br`
-   **Senha:** `Admin@123456`


### Railway (produção)

No Railway, as migrações (`prisma migrate deploy`) **não executam seed automaticamente**.

Para garantir a criação do usuário admin e perfis iniciais, há duas opções:

1. **Executar seed manualmente (uma vez)** após o primeiro deploy:

```bash
pnpm db:seed
```

2. **Executar seed automaticamente no start** definindo a variável de ambiente:

```env
RUN_DB_SEED=true
```

> Recomendação: após popular o banco inicial, altere para `RUN_DB_SEED=false` (ou remova), embora o seed atual use `upsert` e seja idempotente para usuários/roles padrão.


### Troubleshooting Railway (API indisponível)

Se os logs mostram que a aplicação subiu, mas o Railway retorna **"Application failed to respond"**, verifique:

1. Se a aplicação está escutando em `0.0.0.0` (não apenas `localhost`).
2. Se a variável `PORT` do Railway está sendo respeitada.
3. URL correta da API: `https://<seu-dominio>.up.railway.app/api/v1`.

O backend já está configurado para usar:

- `HOST=0.0.0.0` (padrão)
- `PORT` vindo do ambiente

### 3. Ambiente de Produção

Para simular o ambiente de produção com Nginx como proxy reverso, execute:

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build -d
```

Nesta configuração, todos os serviços são acessados através do Nginx nas portas `80` e `443`. O acesso direto aos contêineres do frontend e backend é desativado por segurança.

## Funcionalidades e Segurança

### Autenticação e Autorização

-   **JWT (Access + Refresh Tokens):** Sistema de autenticação seguro que minimiza a exposição de tokens de acesso.
-   **RBAC (Role-Based Access Control):** O acesso a rotas e funcionalidades é controlado por perfis (`ADMIN`, `DIRECTOR`, `MEMBER`).
-   **Guards de Rota:** Tanto no backend (NestJS Guards) quanto no frontend (Next.js Middleware), garantindo que apenas usuários autorizados acessem áreas restritas.

### Segurança

-   **Hashing de Senhas:** Senhas são armazenadas com hash usando `bcrypt`.
-   **Validação de Dados (DTOs):** `class-validator` e `class-transformer` no backend para garantir a integridade dos dados recebidos.
-   **Proteção contra Vulnerabilidades Comuns:** Uso de `helmet` para proteção contra ataques como XSS, e configuração de CORS.
-   **Rate Limiting:** Limita o número de requisições para proteger a API contra ataques de força bruta.
-   **Logs de Auditoria:** Ações críticas (criação, edição, exclusão) são registradas para fins de auditoria.
-   **Upload Seguro:** Validação de tipo e tamanho de arquivo, com Cloudinary para produção ou armazenamento local para desenvolvimento.

## Scripts Úteis

Os seguintes comandos podem ser executados dentro dos contêineres ou localmente se o ambiente estiver configurado.

### Backend

```bash
# Acessar o contêiner do backend
docker exec -it ajucat_backend sh

# Gerar cliente Prisma
pnpm exec prisma generate

# Aplicar migrações do Prisma
pnpm exec prisma migrate deploy

# Executar o seed do banco de dados
pnpm exec prisma db seed
```

### Frontend

```bash
# Acessar o contêiner do frontend
docker exec -it ajucat_frontend sh

# Instalar dependências (se necessário)
pnpm install
```
